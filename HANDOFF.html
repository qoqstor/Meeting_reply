📋功能概覽

sei_reply_assistant.py 是一個獨立的 Python tkinter GUI 工具，掛載在 jt-live-whisper 旁邊執行，提供：

  • 即時翻譯顯示 — 監控 jt-live-whisper log，左右並列顯示日文原文 / 中文翻譯（自動捲動）
  • AI 回覆建議（日文） — 取最近 30 句 / 10 分鐘對話，送進 AI 生成日文回覆
  • 中文草稿（雙向回覆） — AI 同步輸出中文草稿，方便中文使用者理解並回應 v2
  • 自動觸發模式 — 偵測句尾標點（。！？）靜默 8 秒後自動生成 v2
  • 對話歷史記憶 — 多輪對話送進 AI，保留最多 10 輪歷史 v2
  • TTS 播放 — 生成後可朗讀日文回覆或中文草稿（edge-tts）v2
  • 匯出逐字稿 — 一鍵匯出 Excel / Word / TXT，含原文、翻譯、AI 回覆 v2
  • 可調整視窗比例 — 上下兩區 PanedWindow，拖曳 sash 自由調整比例 v2.1
  • 多 AI Provider — Ollama（本地免費）、Anthropic Claude、Google Gemini
  • 可設定情境 Profile — profiles/*.json 儲存每個專案的 system prompt
  • jt-live-whisper 啟動鍵 — 直接從 UI 開啟 / 停止翻譯程式，含模式選單
✅ 工具不依賴 jt-live-whisper 的 --webui 模式；直接讀取 log 檔，CLI 模式也可用。

使用流程（v2）

  1. UI 啟動後選擇情境 Profile → 設定 Provider / Model → 按「連線」
  2. 點擊 🎙 開啟即時翻譯 啟動 jt-live-whisper
  3. 對方說話 → 即時翻譯區自動更新（左=日文，右=中文）
  4. 點擊綠色 ▶ 生成回覆建議（或啟用「自動觸發」checkbox）
  5. AI 輸出 日文回覆（左窗，可直接複製給對方）+ 中文草稿（右窗，理解用）
  6. 點「▶ TTS」朗讀，或「複製」貼至對話視窗
  7. 點「匯出逐字稿」存成 Excel / Word

監控的 log 格式

jt-live-whisper 依翻譯模式寫出不同格式：

# ja_zh（日中雙向）— 含方向符號 ▶ / ◀
[14:37:42] ◀ [日] 助かりました。じゃあ、ちょっとこの
[14:37:42] ◀ [中] 這就是我們所需要的.

# ja2zh（日翻中）— 無方向符號
[09:43:43] [日] 生機順利完成吊掛22日晚間
[09:43:43] [中] 沒有人知道.

正則表達式（▶◀ 為 optional，v2.1 修正）：

LINE_RE = re.compile(r'^\[(\d{2}:\d{2}:\d{2})\]\s*(?:[▶◀]\s*)?\[([日中])\]\s*(.+)$')
SENTENCE_END_RE = re.compile(r'[。！？!?]')  # v2: 用於自動觸發

_find_latest_log() 使用 *逐字稿_*.txt 涵蓋所有模式，取 mtime 最新檔：

def _find_latest_log() -> str | None:
    files = []
    for d in _jtw_log_dirs():
        files.extend(glob.glob(os.path.join(d, "*逐字稿_*.txt")))  # v2.1: was "日中雙向_逐字稿_*.txt"
    return max(files, key=os.path.getmtime) if files else None

多目錄掃描

def _jtw_log_dirs() -> list[str]:
    dirs = [LOGS_DIR]
    if JTW_EXE.exists():
        jtw_dir = JTW_EXE.parent
        dirs.append(str(jtw_dir / "logs"))
        dirs.append(str(jtw_dir / "_internal" / "logs"))
    return list(dict.fromkeys(dirs))

⚡自動觸發模式

工作原理

# log_monitor：偵測句尾，設 flag
_ui_queue.put({
    "type": "transcript_add",
    "ts": ts, "ja": text, "zh": zh,
    "has_sentence_end": bool(SENTENCE_END_RE.search(text))  ← 新增
})

# _poll()：收到 flag 後記錄時間
if msg.get("has_sentence_end") and self._auto_reply_var.get():
    self._last_sentence_end = time.monotonic()

# _check_auto_trigger()：每 500ms 輪詢
if (self._auto_reply_var.get()
        and self._last_sentence_end > 0
        and not self._generating
        and time.monotonic() - self._last_sentence_end >= AUTO_TRIGGER_DEBOUNCE):
    self._last_sentence_end = 0.0  ← 重置，避免重複觸發
    self._generate()
ℹ AUTO_TRIGGER_DEBOUNCE = 8.0 秒（v2.1 從 3.0 調整）。偵測到句尾後，若 8 秒內有新內容到達，計時器被後者覆蓋，只有真正靜默 8 秒才觸發。給 Ollama 本機 LLM 足夠時間跑完上一次。

↔️雙向回覆

輸出格式

AI 被要求以 JSON 格式輸出：

{"ja_reply": "日本語の返答（3〜4文）", "zh_reply": "中文草稿（對應翻譯）"}

解析邏輯（_parse_bilingual）

def _parse_bilingual(text: str) -> tuple[str, str]:
    try:
        m = re.search(r'\{[^{}]*"ja_reply"[^{}]*\}', text, re.DOTALL)
        if m:
            data = json.loads(m.group())
            return data.get("ja_reply", "").strip(), data.get("zh_reply", "").strip()
    except (json.JSONDecodeError, AttributeError):
        pass
    return text.strip(), ""  # fallback: 全文 as ja_reply
⚠ Ollama streaming 時，reply_chunk 事件顯示的是原始累積文字（JSON 尚未完成），reply_done 後才解析並更新雙欄。這是設計意圖，不是 bug。

play_tts() 機制

async def _tts_save_async(text, voice) -> str:
    tmp = tempfile.NamedTemporaryFile(suffix=".mp3", delete=False)
    communicate = _edge_tts.Communicate(text, voice)
    await communicate.save(tmp.name)
    return tmp.name

def play_tts(text, lang="ja"):
    if not _tts_lock.acquire(blocking=False):
        return  # skip if already playing
    try:
        loop = asyncio.new_event_loop()
        tmp_path = loop.run_until_complete(_tts_save_async(text, voice))
        loop.close()
        os.startfile(tmp_path)
        time.sleep(90); os.unlink(tmp_path)
    finally:
        _tts_lock.release()
ℹ _tts_lock 使用 blocking=False，若已有 TTS 播放中，新的呼叫直接略過（不排隊）。

🔧v2.1.0 異動記錄（2026-06-25）

BUG FIX
非雙向模式（ja2zh 等）監控來源指向昨日舊檔
根因：_find_latest_log() glob 寫死 日中雙向_逐字稿_*.txt，只找 ja_zh 模式。 ja2zh 模式寫出 日翻中_逐字稿_*.txt，找不到當日檔 → 落回 mtime 最近的舊檔（昨日）。
修法：glob 改為 *逐字稿_*.txt，涵蓋所有翻譯模式。
BUG FIX
非雙向模式翻譯內容不顯示（LINE_RE 不匹配）
根因：LINE_RE 強制要求 [▶◀] 方向符號。雙向模式（ja_zh）有此符號； 單向模式（ja2zh 等）格式為 [HH:MM:SS] [日] text，無方向符號 → regex 不匹配 → 解析 0 筆。
修法：[▶◀] 改為 optional group (?:[▶◀]\s*)?，向下相容兩種格式。
FEATURE
上下可調整版面（ttk.PanedWindow）
即時翻譯區與建議回覆區改用 ttk.PanedWindow(orient="vertical")。 拖曳 sash 可自由調整上下比例（預設 weight=3:2）。 建議回覆移除固定 height=4，隨 pane 伸縮。
TUNING
MAX_SENTENCES 7 → 30，AUTO_TRIGGER_DEBOUNCE 3.0 → 8.0
取用更完整的對話脈絡（最近 30 句 / 10 分鐘）。 自動觸發 debounce 從 3 秒調整為 8 秒，讓本機 Ollama LLM 有足夠時間完成上一次生成再觸發下一次。

🐛已知問題

ja2zh 等單向模式翻譯不顯示

已於 v2.1.0 修正。根因：LINE_RE 強制 [▶◀] + _find_latest_log() glob 只找 日中雙向_*。FIXED v2.1

transcript_update_zh 行號計算

line_no * 3 - 1 假設每筆記錄精確佔 3 行。若日文原文換行，行號偏移導致「…」替換失敗。BUG

修法方向： 改用 tkinter Text widget 的 mark 或 tag 機制標記每筆記錄位置。

Ollama 首次冷啟動

qwen2.5:7b 首次載入需 30-60s，看似 timeout。第二次起快很多。WARN

TTS 暫存檔清除時機

TTS 音訊存為 temp MP3，os.startfile() 後 sleep(90) 再刪除。若音訊超過 90 秒，刪除時可能仍在播放（Windows 會拒絕刪除）。靜默忽略 OSError。WARN

AI 未返回 JSON 時 zh_reply 為空

_parse_bilingual() fallback 時 zh_reply=""，中文草稿欄位顯示空白。可透過調整 system_prompt 提高 JSON 輸出成功率。TODO
